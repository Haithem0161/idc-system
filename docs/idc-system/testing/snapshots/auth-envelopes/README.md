# Phase-02 Auth + Users + Settings IPC Envelope Snapshots

Canonical wire-format samples for the phase-02 IPC commands. The frontend's Zod schemas under `src/lib/schemas/` MUST accept these envelopes byte-for-byte (after canonicalisation). The Rust `#[derive(Serialize)]` paths under `src-tauri/src/domains/{auth,settings}/commands.rs` MUST produce envelopes equivalent under canonical-JSON comparison.

| File | Source | Validated by |
|-|-|-|
| `login-result.json` | `auth_login` IPC response (`LoginResult` struct). | `src/lib/schemas/ipc-contract-phase02.test.ts::LoginResult shape`; structural pin only -- the `id` and timestamps are placeholders. |
| `user-response.json` | `users_list` / `users_get` / `users_create` / `users_update` IPC response (`UserResponse` struct). | `src/lib/schemas/ipc-contract-phase02.test.ts`; the CRITICAL invariant is no `password_hash` key at any depth. |
| `setting-response.json` | `settings_list` / `settings_get` / `settings_update` IPC response (`SettingResponse` struct with tagged `SettingValue`). | `src/lib/schemas/setting.test.ts::SettingSchema parses a full setting row`; the `value` field is a discriminated union on `valueType`. |

## Schema invariants pinned by these snapshots

- **`UserResponse` carries no `password_hash` at any nesting depth.** Verified separately by `cargo test --test users_phase02 -- users_list_response_strips_password_hash_at_serde_boundary` (Rust side) and by `auth.queries.test.ts::useUsersList response never carries password_hash` (TS side).
- **`SettingValue` uses Serde's adjacently-tagged enum representation** (`tag = "valueType", content = "value"`). The four legal `valueType` values are `int | decimal | text | bool`; no fifth tag is permitted by either the Rust `SettingValue` enum or the TS `SettingValueSchema`.
- **`LoginResult.mode` is one of `online | offline`** (lowercase, from `LoginMode` enum `#[serde(rename_all = "lowercase")]`).
- **`UserResponse.role` is one of `superadmin | receptionist | accountant`** (lowercase from `UserRole` enum).

## Regeneration policy

Per `.claude/rules/testing.md` §10: regenerating any snapshot requires:

1. Explicit `--update-snapshots` flag (no auto-accept).
2. Human reviewer in the PR.
3. Written justification in the PR body.

These files are STRUCTURAL pins. The `id` and timestamp fields are placeholders -- the validating tests check shape only, not exact values. If the SHAPE changes (a field added, removed, or renamed; an enum value added; a tag changed), both the matching Rust integration test and the TS contract test fail simultaneously, which is the desired alarm.
